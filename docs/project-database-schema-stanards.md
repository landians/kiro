# Kiro 数据库 Schema 设计规范

## 1. 文档范围

本文档用于整理 PostgreSQL 数据库在 schema 设计、命名、主键、外键、索引、分区和结构演进方面的约束。内容主要参考阿里云文章《PostgreSQL 数据库开发规范》，并结合当前项目已经确定的 PostgreSQL 使用约束进行收敛，只保留与表结构设计强相关的部分，不展开查询写法和运维管理细则。

## 2. 设计目标

- 表结构优先保证一致性、可读性、可维护性。
- 相同业务语义在不同表中保持统一命名和统一类型，避免跨表 join、迁移和程序映射时出现隐性错误。
- 核心业务数据优先使用结构化字段表达，不把可以建模的关系全部塞进非结构化列。
- 所有 schema 设计都应兼顾索引可用性、更新成本、历史归档和后续迁移难度。

## 3. Schema 与命名规范

### 3.1 通用命名规则

- 数据库对象名长度必须控制在 PostgreSQL 标识符限制内，建议库名、表名、字段名总长度不超过 63 个字符。
- 表名、字段名、索引名、约束名、视图名、函数名、序列名统一只使用小写字母、数字和下划线。
- 对象名禁止以数字开头，禁止以 `pg` 开头，禁止使用 PostgreSQL 保留字。
- SQL 查询中的别名也遵循同样的命名规则，不使用中文、空格或特殊符号。

### 3.2 约定式前缀

- 主键索引统一以 `pk_` 开头。
- 唯一索引统一以 `uk_` 开头。
- 普通索引统一以 `idx_` 开头。
- 临时表统一以 `tmp_` 开头。
- 分区子表统一使用可推断后缀，例如按年分区可使用 `table_name_2026` 这类形式。

### 3.3 Schema 组织规则

- 不将所有业务对象默认堆放在 `public` schema。
- 每个应用或独立业务域应分配明确的 schema，名称最好与应用名或业务名保持一致，便于权限划分和问题定位。
- 数据库名、schema 名、应用名应尽量保持一一对应或具备稳定映射关系。

### 3.4 注释规范

- 源文章建议 comment 尽量避免中文，原因是跨客户端、备份和导出时可能出现编码不一致。
- 本项目如果编写数据库注释，默认优先使用英文或稳定的中英双语短句；若使用中文，数据库、驱动、导出工具必须统一使用 UTF-8，避免 `pg_dump` 与恢复时产生乱码。

## 4. 字段设计规范

### 4.1 同语义字段必须统一

- 多张表中的相同业务字段必须保持列名一致、数据类型一致、精度一致。
- 同一语义字段的可空性、默认值和校对规则也应尽量保持一致，避免上层程序出现额外的兼容分支。
- 应用程序中的字段类型定义必须与数据库定义一致，避免因为隐式类型转换导致索引失效、比较异常或运行时错误。

例如：

- `user_id` 在所有业务表中都应保持同一种整数类型，不能在某些表中使用 `bigint`，另一些表中使用 `varchar`。
- 时间字段应统一使用同一种时间语义，例如统一使用 `timestamptz` 表达带时区时间点。

### 4.2 主键与标识字段

- PostgreSQL 业务表主键统一使用数据库自增主键，推荐 `bigint generated always as identity`。
- 外键字段类型必须与目标主键完全一致，统一使用 `bigint`。
- 不在核心业务表中混用 `uuid` 主键和自增主键，避免 schema 风格不一致和 join 成本增加。
- 自增主键只承担内部关系标识职责；如需要对外暴露稳定编号，应额外设计业务编码字段，例如 `order_code`、`user_code`。

### 4.3 字符与长度设计

- 默认统一使用 UTF-8 编码。
- 设计文本字段时要区分字符长度和字节长度，尤其是在多语言、中文或 emoji 场景下。
- 需要参与索引设计时，应优先关注字节长度，而不是只看字符个数。

### 4.4 高变字段与扩展属性

- 表结构应尽量提前规划，避免频繁执行加列、改类型、改长度等可能触发表重写的操作。
- 对于确实难以提前稳定建模、并且字段扩展频繁的扩展信息，可使用 `jsonb` 承载非核心属性。
- `jsonb` 只用于扩展属性，不应替代核心关系字段、状态字段、主外键或高频过滤条件。

## 5. 约束与关联规范

### 5.1 外键约束

- 使用外键时，必须显式声明 `ON DELETE` 和 `ON UPDATE` 行为，不能把级联策略留给默认值碰运气。
- 具体动作根据业务语义选择 `cascade`、`set null`、`set default` 或 `restrict`，但必须在 DDL 中写明。
- 若业务需要保留历史记录而不是级联清理，优先使用显式拒绝删除或软删除策略，而不是默认级联。

### 5.2 外键索引

- PostgreSQL 不会自动为外键列创建索引，因此所有高频关联字段默认都要补充索引。
- 未为外键列建立索引时，被引用表执行更新或删除可能退化为低效扫描，影响写入和锁等待。
- 多列外键应根据查询路径建立对应联合索引，而不是只机械地给首列建单列索引。

### 5.3 其他约束

- 需要保证分区表主键全局唯一时，应在设计阶段统一规划序列策略，避免不同分区各自生成重复 ID。
- 可以通过分段序列、不同步长序列或预留 ID 范围等方式实现跨分区唯一性。
- 关键业务表应优先使用数据库约束表达不变量，例如唯一约束、检查约束和非空约束，而不是完全依赖应用层兜底。

## 6. 索引设计规范

### 6.1 基本原则

- 索引应服务于明确的查询路径、排序路径和关联路径，禁止为“可能会用到”而盲目加索引。
- 设计索引时必须同时考虑查询收益、写放大、维护成本和存储占用。

### 6.2 B-Tree 索引限制

- B-Tree 索引字段不建议超过约 2000 字节。
- 对超长文本、摘要内容或大字段，如果必须支持检索，应优先考虑函数索引、哈希后索引、全文检索索引，或改造查询方式。

### 6.3 特殊索引策略

- 固定条件查询可以使用部分索引，减少索引体积并提升命中效率。
- 频繁使用表达式作为查询条件时，应考虑表达式索引或函数索引。
- 范围检索场景优先考虑 PostgreSQL 范围类型配合 GIST 索引。
- 中文分词检索、全文检索场景可使用 GIN 类索引方案。
- 模糊匹配、相似度匹配场景应基于明确需求选择合适索引；没有针对性索引时，不应放行高成本的全模糊扫描。

### 6.4 高频更新表

- 对频繁更新的表，可在建表时设置较低的 `fillfactor`，例如 `fillfactor = 85`，为 HOT 更新预留页内空间。
- 此类设置应根据真实更新模式和表膨胀情况决定，不能无差别应用到所有表。

## 7. 分区与生命周期规范

### 7.1 何时考虑分区

- 对于超大表、时间序列型表、冷热数据明显分层的表，应在 schema 设计阶段提前考虑分区。
- 文章给出的经验阈值是：大致超过 8GB 或超过 1000 万行的高频访问大表，可以评估分区。
- 分区的目标不仅是查询提速，也包括提升更新效率、归档效率、备份恢复效率和索引维护效率。

### 7.2 时间分区

- 具有明确历史保留周期的业务表优先按时间分区。
- 清理历史数据时，优先使用 `DROP PARTITION`、`TRUNCATE` 等结构化方式，而不是对海量旧数据直接执行 `DELETE`。
- 分区键要与主要查询条件和数据生命周期一致，避免只做形式上的分区。

### 7.3 分区唯一性

- 分区表若仍要求全局唯一主键，需要在序列、约束和路由策略上提前统一设计。
- 不允许等到数据量上来之后再临时修补“不同分区 ID 可能重复”的问题。

## 8. 项目落地约束

结合当前项目既有规范，新增或修改表结构时还必须满足以下约束：

- 所有业务表主键统一为 `bigint generated always as identity`。
- 所有关联字段统一使用 `bigint`，并与目标主键类型严格一致。
- 业务表中的相同语义字段必须保持统一命名，例如 `user_id`、`created_at`、`updated_at` 不能出现多个变体。
- 若一个字段需要被 join、过滤、排序或唯一识别，就优先建模为显式列，而不是藏在 `jsonb` 中。
- 涉及历史归档、日志、事件、审计等高增长表时，必须在建表阶段说明是否需要分区和历史清理策略。

## 9. Schema 变更检查清单

每次提交 migration 前，至少检查以下问题：

- 命名是否符合小写下划线规则，且没有使用保留字。
- 同语义字段是否与现有表保持名称、类型、精度一致。
- 主键是否使用统一的 `bigint identity` 方案。
- 外键是否声明了明确动作，并为外键列补充了合适索引。
- 需要索引的长文本字段是否误用了普通 B-Tree。
- 高频更新表是否需要单独设置 `fillfactor`。
- 大表是否需要分区，历史删除是否可通过分区裁剪完成。
- 新增扩展字段是否真的应该落到 `jsonb`，还是应该建成结构化列。
- 注释、默认值、非空约束、唯一约束是否已经表达清楚，而不是依赖应用层约定。
