# Kiro 项目计划总览

## 1. 项目总体定位
Kiro 是一个面向订阅制数字商品或服务的后端基础设施框架，目标是在有新产品想法时，可以基于统一底座快速搭建出一个具备账号、认证、JWT 鉴权、管理后台、商品、支付、订阅、通知和本地化能力的后端服务。

## 4. 功能模块划分

### 4.1 核心模块范围

项目当前规划包含以下十个功能模块：

1. 账号模块。
2. 登录模块：支持 Google 登录。
3. 管理后台模块：支持多个管理员，通过统一后台权限校验访问后台能力。
4. 商品模块：定义售卖商品、计划和权益。
5. 支付模块：支持 Stripe 和 Creem。
6. 订阅模块。
7. 邮件模块。
8. 通知模块。
9. 多语言和本地化模块。
10. 邀请模块。

### 4.2 模块优先级建议

为保证项目尽快形成可用闭环，按以下顺序推进：

- P0：账号模块、登录模块、管理后台基础权限、商品模块、基础可观测、基础错误模型。
- P1：支付模块、订阅模块、管理后台业务操作面板。
- P2：邮件模块、通知模块、多语言和本地化模块。
- P3：邀请模块、模块联调、运维治理、回溯审计、性能优化。

## 5. 项目开发里程碑

整体推进原则：

- 先完成基础底座，再形成从登录到支付再到订阅的主业务闭环，最后补齐异步能力、增长能力和治理能力。
- 每个大里程碑都必须能单独合并、单独部署到开发环境、单独回归验证。
- 每个小里程碑都必须具备“实现完成后可独立测试”的交付边界，避免做出一半才能验证的长链路任务。
- 管理后台默认采用“管理员即拥有全部后台权限”的简单模型，先不引入细粒度 RBAC。

### 5.1 M0：基础底座与运行时闭环

目标：

- 建立可启动、可观测、可配置、可迁移的后端底座。
- 在不引入复杂业务模块前，先把工程骨架、健康检查、数据库迁移、统一错误模型和鉴权基础能力铺好。

#### M0.1 应用启动骨架与统一配置

- 实现 `main.rs` 启动流程、配置对象、日志初始化、环境变量加载、优雅退出和依赖装配骨架。
- 补充基础路由：`/health/live`、`/health/ready`。
- 独立测试：应用可正常启动；健康检查接口返回稳定响应；缺失关键配置时启动失败且日志可定位原因。

#### M0.2 PostgreSQL / Redis 接入与迁移基线

- 建立 PostgreSQL 连接池、Redis 客户端和 migration 执行流程。
- 增加第一批基础表结构迁移文件，并确保符合数据库 schema 设计规范。
- 独立测试：空库可执行全量 migration；应用启动时可校验数据库和 Redis 连通性；连接失败时 `ready` 检查返回失败。

#### M0.3 统一响应模型、错误模型与 `X-Trace-Id`

- 建立统一的成功响应结构、错误响应结构、错误码映射和请求级 trace 传递机制。
- 所有 HTTP 响应和错误响应都包含 `X-Trace-Id`。
- 独立测试：正常请求和异常请求都能返回一致结构；日志、响应头和错误响应中的 trace 可关联。

#### M0.4 JWT / Refresh Token 基础设施与鉴权中间件

- 完成 JWT 签发、解析、刷新、显式登出吊销、`ua_hash` 绑定、Redis 持久化黑名单抽象接口与鉴权中间件骨架。
- 暂不依赖完整账号体系，先把 token 基础能力做成独立模块。
- 独立测试：可签发并验证 token；刷新和显式登出可以闭合 token 生命周期；无效签名、过期 token、`ua_hash` 不匹配时可稳定拒绝；黑名单命中时请求被拦截。

#### M0.5 基础测试框架与可观测基线

- 建立单元测试、集成测试、接口契约测试的基本目录和约定。
- 为关键启动链路、数据库依赖、HTTP 基础行为补齐最小自动化测试。
- 独立测试：执行测试命令可覆盖健康检查、配置校验、token 校验和基础错误返回。

### 5.2 M1：账号与登录闭环

- 建立用户、身份绑定、登录、刷新、登出和当前用户信息获取的最小可用闭环。
- 形成第一个可对外使用的业务能力集合。

#### M1.1 账号领域模型与持久化

- 设计并落地用户表、用户状态字段、审计字段和基础仓储实现。
- 明确账号创建、禁用、恢复等领域状态约束。
- 独立测试：可创建用户、查询用户、更新用户状态；非法状态迁移会被拒绝。

#### M1.2 Google 登录接入与身份绑定

- 接入 Google 登录回调流程，建立第三方身份绑定表与适配器。
- 处理首次登录创建账号、再次登录绑定已有账号、重复绑定冲突等场景。
- 独立测试：模拟 Google 登录成功与失败路径；首次登录可创建用户；重复绑定冲突有明确错误返回。

#### M1.3 登录、刷新、登出、个人页 接口

- 暴露登录结果接口、刷新接口、登出接口和当前用户信息接口。
- 将 JWT / Refresh Token 基础设施与账号领域串起来。
- 独立测试：登录后可访问受保护接口；刷新后旧 token 按策略失效；登出后 token 命中黑名单。

#### M1.4 账号安全治理与限流基线

- 为登录、刷新、登出路径增加最小限流、审计日志和异常行为记录。
- 把登录相关关键事件纳入统一可观测字段。
- 独立测试：高频刷新、异常 `ua_hash`、重复登出等场景可稳定处理并产生日志。

### 5.3 M2：管理后台与管理员权限闭环

目标：

- 建立独立的管理后台能力入口，支持多个管理员账号并通过统一后台权限校验访问后台接口。
- 当前阶段先采用简单权限模型：只要是有效管理员，就可以操作全部管理后台功能，不做细粒度角色拆分。

#### M2.1 管理员数据模型与授予机制

- 设计管理员表、管理员状态字段和授予关系，允许将多个普通用户账号提升为管理员。
- 管理员身份以独立后台授权关系表达，不直接把后台权限硬编码在普通用户表里。
- 独立测试：可以授予多个管理员；被禁用或撤销的管理员无法再通过后台权限校验；同一用户不会被重复创建管理员关系。

#### M2.2 管理后台鉴权与统一权限中间件

- 增加后台路由分组，例如 `/admin/*`，并通过统一中间件校验“当前登录用户是否为有效管理员”。
- 后台权限模型暂时只判断是否为管理员，不区分模块级权限和操作级权限。
- 独立测试：普通用户访问后台接口被拒绝；有效管理员可访问后台接口；管理员失效后旧 token 在后台接口中被正确拦截。

#### M2.3 管理员管理接口与后台审计日志

- 提供最小后台能力：管理员列表、授予管理员、停用管理员、恢复管理员。
- 增加后台操作审计日志，记录操作人、模块、动作、目标对象和结果状态。
- 独立测试：管理员增删改查路径可正常工作；每次后台操作都能写入审计记录；失败路径也能记录关键信息。

#### M2.4 后台基础控制面板

- 提供后台首页所需的最小聚合接口，例如用户数、商品数、支付订单数、订阅数等基础概览。
- 作为后续商品、支付、订阅后台运营入口的统一承载层。
- 独立测试：管理员可以获取概览数据；普通用户无法访问；在各业务表为空时也能返回稳定结构。

### 5.4 M3：商品与权益建模闭环

目标：

- 建立商品、套餐、价格、权益的统一模型，为支付和订阅提供稳定商品底座。
- 让商品模块可以在不依赖支付模块的情况下独立完成建模与查询。

#### M3.1 商品、计划、权益 Schema 与仓储实现

- 设计商品表、计划表、价格表、权益表及必要关联表。
- 明确启用状态、展示状态、排序字段和多语言展示位的建模方式。
- 独立测试：可创建和查询商品、计划与权益关系；禁用商品不会出现在公开查询中。

#### M3.2 商品查询接口与对外展示 DTO

- 提供商品列表、商品详情、套餐列表等读取接口。
- 保证接口层只负责组装，不把业务筛选逻辑散落在 handler 中。
- 独立测试：接口返回稳定 DTO；上下架状态、排序规则和过滤条件符合预期。

#### M3.3 购买前置校验能力

- 建立“商品是否可售”“计划是否有效”“价格是否可下单”等前置校验服务。
- 为后续支付创建统一的购买入口校验逻辑。
- 独立测试：无效商品、无效价格、已下架计划会被一致拒绝；合法商品可通过校验。

### 5.5 M4：支付下单与回调闭环

目标：

- 打通从商品选择到支付发起、支付回调、支付记录入库的完整链路。
- 先以统一支付抽象为核心，再分别接入 Stripe 和 Creem。

#### M4.1 支付领域模型与订单基础结构

- 设计支付订单、支付尝试、支付回调记录、幂等键等核心表结构。
- 明确支付状态机和状态迁移规则。
- 独立测试：可创建待支付订单；重复创建同一幂等键请求时不会生成重复支付记录；非法状态迁移会被拒绝。

#### M4.2 Stripe 支付接入

- 实现 Stripe 适配器、下单能力和回调验签。
- 将第三方差异收敛在 `infrastructure/payments/stripe` 中。
- 独立测试：可创建 Stripe 支付会话；回调验签成功后订单状态更新；重复回调不会重复入账。

#### M4.3 Creem 支付接入

- 以统一支付接口为边界实现 Creem 适配器。
- 保证应用层不需要分支散落判断不同支付提供商细节。
- 独立测试：Creem 下单、回调、失败处理路径可独立跑通；与 Stripe 共存时接口行为保持一致。

#### M4.4 支付幂等、补偿与对账基础

- 为支付回调、超时订单、失败重试建立最小补偿策略。
- 补充支付关键日志、事件投递和手工排障所需字段。
- 独立测试：重复回调、超时回调、部分失败重试场景下系统状态保持一致。

### 5.6 M5：订阅生命周期闭环

目标：

- 建立订阅创建、生效、续费、取消、过期和权益同步的核心业务闭环。
- 让支付完成后可以自然驱动订阅状态变化。

#### M5.1 订阅模型与状态机

- 设计订阅表、订阅周期、续费状态、取消策略和权益关联模型。
- 明确试用、激活、续费失败、到期、取消等状态语义。
- 独立测试：订阅状态迁移符合业务规则；无效迁移被拒绝；周期边界计算正确。

#### M5.2 支付成功驱动订阅开通

- 将支付完成事件与订阅激活流程打通。
- 建立支付订单与订阅实例的关联关系。
- 独立测试：支付成功后自动创建或激活订阅；重复处理同一支付事件不会生成重复订阅。

#### M5.3 续费、取消、过期处理

- 实现主动取消、自动续费、续费失败、自然到期等处理逻辑。
- 预留定时任务或异步事件入口用于后续批处理。
- 独立测试：取消后状态正确；到期后权益失效；续费失败不会误发放权益。

#### M5.4 权益判定与受保护资源校验

- 提供“用户当前是否具备某项权益”的统一查询能力。
- 为未来付费接口或资源下载接口提供授权依据。
- 独立测试：用户在不同订阅状态下的权益判定结果正确，且接口可复用。

### 5.7 M6：邮件、通知与多语言异步闭环

目标：

- 建立事件驱动的异步发送能力，补齐支付和订阅后的用户触达能力。
- 让邮件、站内通知和本地化能力既可独立验证，又能组合进主链路。

#### M6.1 Redis Stream 事件总线与消费框架

- 建立领域事件投递、消费、重试和失败观测基础设施。
- 统一事件格式、幂等键和消费日志字段。
- 独立测试：事件可被投递和消费；重复消费不会重复执行；失败事件可被重试并可观测。

#### M6.2 邮件模块

- 实现邮件模板、邮件发送适配器和关键邮件场景，如欢迎邮件、支付成功邮件、订阅变更邮件。
- 邮件发送必须走异步链路，不阻塞主业务请求。
- 独立测试：事件触发后可生成正确邮件任务；模板变量渲染正确；发送失败可重试。

#### M6.3 通知模块

- 实现站内通知或通用通知模型，支持投递、已读状态和基础查询。
- 把支付、订阅、邀请等关键事件统一投递到通知系统。
- 独立测试：通知可创建、查询和标记已读；重复事件不会产生重复通知。

#### M6.4 多语言与本地化模块

- 建立多语言文案资源、语言选择策略和模板本地化能力。
- 商品展示文案、邮件文案、通知文案都通过统一本地化接口获取。
- 独立测试：在不同 locale 下返回正确文案；缺失翻译时有稳定回退策略。

### 5.8 M7：邀请体系与系统稳定化

目标：

- 补齐增长能力与治理能力，让项目从“可用”进入“可上线维护”状态。
- 把邀请链路、审计、运维和性能治理收口到一个可发布阶段。

#### M7.1 邀请模型与邀请发起能力

- 设计邀请关系、邀请码或邀请链接、邀请状态和奖励归因模型。
- 提供邀请创建与邀请查询接口。
- 独立测试：可创建邀请；重复邀请策略符合预期；非法邀请参数被拒绝。

#### M7.2 邀请接受与奖励归因

- 打通邀请接受、账号关联、首单归因或奖励触发逻辑。
- 保证邀请链路具备幂等保护。
- 独立测试：同一邀请不会被重复消费；邀请人与被邀请人关系正确落库；奖励只发放一次。

#### M7.3 审计、补偿与运维治理

- 为支付、订阅、邀请、邮件、通知建立必要审计日志和补偿入口。
- 梳理死信、失败重试、人工修复和排障操作手册所需的数据支撑。
- 独立测试：关键失败场景可通过补偿恢复；审计数据足够定位链路问题。

#### M7.4 发布前稳定性与性能验收

- 完成关键接口压测、超时治理、连接池配置、慢查询排查和安全自查。
- 补齐回归测试清单和发布检查清单。
- 独立测试：核心接口在目标并发下无明显错误；关键路径日志、指标和追踪字段完整；主要业务链路回归通过。

## 6. 里程碑验收原则

- 大里程碑验收必须以“真实可运行闭环”为标准，而不是只看代码文件是否存在。
- 小里程碑验收必须至少包含一种自动化测试，推荐组合：领域单元测试 + 仓储集成测试 + HTTP 契约测试。
- 涉及第三方系统的里程碑，必须同时验证成功路径、失败路径、超时路径和重复回调路径。
- 每完成一个大里程碑，都应回写相关设计文档、接口约定和迁移说明，避免计划与实现脱节。
